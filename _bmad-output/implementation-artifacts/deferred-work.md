## Deferred from: implementation of 11-1-server-authoritative-sql-generation

- **Self-join table aliasing in `DatasetSqlGenerator`** — When the same table appears twice as two separate canvas nodes (self-join), the generator needs per-node numeric aliases (`"orders" t0`, `"orders" t1`) in FROM, JOIN, SELECT, WHERE, and ORDER BY. The current generator assumes each table appears once. A follow-on story should add this when self-join canvas scenarios are tested. [`DatasetSqlGenerator.cs`]
- **Filter condition with blank `tableName`/`columnName` skipped silently** — `FilterConditionDto` with an empty `tableName`/`columnName` is skipped in WHERE generation rather than returning a validation error. Story 11.2 should add a pre-flight gate: "All filter conditions must reference a table and column." Same root cause as the 10-6/10-7 degenerate-condition defers. [`DatasetSqlGenerator.cs`]
- **`ParameterizedSql` + `Parameters[]` generated but not yet consumed** — `SqlGenerationResult.ParameterizedSql` and `Parameters` are produced for the preview path but nothing reads them yet. Story 11.3 wires these into POST /api/datasets/preview. [`DatasetEndpoints.cs`]
- **`CreateAsync` not wired for builder-mode creates** — `DatasetService.CreateAsync` still treats `is_custom_query = false` as a placeholder-only create (no builder_state parsing). Story 11.2 should add `BuilderState` to `CreateDatasetRequest` and run the generator in `CreateAsync`. [`DatasetService.cs`, `Dtos/CreateDatasetRequest.cs`]
- **`builder_state` normalization deferred to Story 11.2** — `BuilderStateSerializer.Deserialize` returns `null` on invalid JSON, and `DatasetSqlGenerator` defensively null-coalesces missing top-level lists; but full field-by-field normalization (both the frontend `parseBuilderState` and the backend DTO) is still owed. Story 11.2 should complete it. [`builderState.ts`, `BuilderStateDto.cs`]
- **CASE operand/then/else run through `ExpressionSecurityValidator` (wrap-parse)** — Per the story spec, non-empty CASE operand/THEN/ELSE values are validated as expressions before being rendered as quoted literals. A literal value containing an apostrophe or unbalanced quote (e.g. `O'Brien` as a CASE operand) would fail the wrap-parse even though it renders safely as an escaped literal. Revisit when CASE quoting is hardened; filter-condition literals (the common path) are unaffected. [`DatasetSqlGenerator.cs`]

## Deferred from: code review of 11-1-server-authoritative-sql-generation (2026-06-04)

- **Builder-mode save with empty/whitespace `builder_state` skips ALL SELECT-only enforcement** — When `effectiveIsCustomQuery == false` but `effectiveBuilderState` is empty/whitespace, the generator block is skipped (`DatasetService.cs:332-347`) and the custom-query checkpoint (a) is also skipped (guarded by `effectiveIsCustomQuery`), so a stale `current.Query` flows into `effectiveViewQuery` → `CREATE OR REPLACE VIEW` with no SELECT-only enforcement on this path. **Deferred (reason):** bounded risk — `current.Query` was already SELECT-only-enforced on its own write; fold the empty-state guard into Story 11.2's `parseBuilderState`/save-gate normalization pass. [`DatasetService.cs`] (edge)
- **GROUP BY omits CASE / calculated columns** — When any aggregate is present, GROUP BY is built only from checked non-aggregated columns (`DatasetSqlGenerator.cs:177-190`); CASE and calculated columns added to the SELECT list are not added to GROUP BY, so a mixed aggregate + CASE/calc query produces a PostgreSQL "column must appear in the GROUP BY clause" error at VIEW creation. Fails safe (no corruption; the save errors). A follow-on should add the CASE/calc expression to GROUP BY or document the limitation. [`DatasetSqlGenerator.cs`] (blind)
- **`JoinEdgeDataDto` drops `sourceColumn`/`targetColumn`** — The C# DTO only carries `JoinType`; the JOIN `ON` clause is built from edge-level `SourceHandle`/`TargetHandle` (`DatasetSqlGenerator.cs:122-123`). The canonical `builderState.ts` `JoinEdgeData` contract has `{ sourceColumn, targetColumn, joinType }`. If a client ever sends the join column only in `data.sourceColumn` (per the typed contract) and not in the handle, the JOIN is built from the wrong/empty identifier. Violates the "mirror both files 1:1" rule (Decision 6.11). Align the DTO with the contract in Story 11.2. [`Dtos/BuilderStateDto.cs`] (edge)
- **ORDER BY / CASE WHEN columns not validated against canvas or allowlist** — `ob.TableName`/`ob.ColumnName` (`:200-206`) and CASE WHEN `ColumnName` (`:428`) are emitted as quoted identifiers with no check that they belong to a node on the canvas or are catalog/allowlist columns; `LookupPgType` silently defaults unknown columns to `text` (`:494-496`). No injection (all `Q`-quoted, bounded to allowlisted tables once the `Q` escaping patch lands), but broader than the UI surface — an ORDER BY on an unexposed column of an allowlisted table is accepted. Add a canvas/allowlist gate in Story 11.2. [`DatasetSqlGenerator.cs`] (blind+edge)
- **Opaque validation messages for empty `IN ()` and zero-WHEN `CASE END`** — An empty IN array (`:322-327`) and a CASE column whose WHEN arms are all null/empty (`:414-478`) emit syntactically invalid SQL that is caught only by Step-10 `SqlSelectEnforcer` with the generic message "Generated SQL is not a valid SELECT." rather than a meaningful "IN list cannot be empty" / "CASE needs ≥1 WHEN." Add pre-flight gates (Story 11.2 owns server-side completeness validation). [`DatasetSqlGenerator.cs`] (edge)
- **Nested-filter converter is O(n²) + recursion bounded only by S.T.Json MaxDepth** — `FilterItemDtoConverter.Read` calls `GetRawText()` and re-`Deserialize`s per node (`Dtos/BuilderStateDto.cs:101-113`); a large/deep filter tree is quadratic in payload size, and `BuildWhereView`/`BuildWhereParameterized` recurse with no explicit depth guard (the default `MaxDepth` 64 throws inside the try/catch → 422). Minor CPU-DoS surface; add an explicit depth/size limit if filter payloads grow. [`Dtos/BuilderStateDto.cs`, `DatasetSqlGenerator.cs`] (blind)
- **String-literal escaping assumes `standard_conforming_strings = on`** — `RenderLiteral` (`:524`) and `RenderCaseValue` (`:489`) escape `'` → `''`, which is safe only under PostgreSQL's default `standard_conforming_strings = on`. Documented hardening note; no action while the cluster runs defaults. Same assumption holds project-wide. [`DatasetSqlGenerator.cs`] (blind)
- **CASE THEN/ELSE/operand literals starting with a DML/DDL keyword over-rejected** — Non-empty CASE operand/THEN/ELSE values pass through `ExpressionSecurityValidator` (`:445,457,470`) before being rendered as quoted literals; a legitimate literal like "Drop-off" or "Delete request" is rejected as a "disallowed keyword." Extends the existing implementation-time defer ("CASE operand/then/else run through ExpressionSecurityValidator"). Revisit when CASE quoting is hardened in Story 11.2. [`DatasetSqlGenerator.cs`] (auditor)
- **AC-5 atomicity and many generator edge cases untested** — No test inspects the actual VIEW definition (`pg_views`/`information_schema.views`) to confirm it equals the stored `query`, and no negative test proves a VIEW-DDL failure rolls back the `query` write — AC-5's "same transaction / in sync" property is only indirectly exercised. Untested generator paths: thousands-separator numeric (`IN (1,000)`), null `Data`/`Columns`→500, empty `IN ()`, `NOT IN`, `LIKE`/`ILIKE`, boolean `t`/`f`, scalar value given to IN, array given to `=`, multi-table GROUP BY across joined nodes. Add when hardening Story 11.1 or in 11.2. [`DatasetBuilderModeTests.cs`, `DatasetSqlGeneratorTests.cs`] (auditor+edge)
