## What Didn't Work

### Critical: No Manual Verification

Stories were accepted when automated tests passed. The application was never verified by running it as a user during Epic 3 (or any prior epic). This was identified explicitly in the Epic 7 retrospective where P1-P7 bugs were found after 42 stories — including designer UI layout broken, table provisioning not working, and browser refresh → 401.

**jukhan's direct feedback:** "It felt frustrating. The verify step should be mandatory."

### Stub-Then-Fix Pattern

- `designerApi.saveVersion` (Story 3.1) was a PUT stub to a URL that didn't exist. Four stories shipped before Story 3.6 replaced it with a real POST. Every save attempt during that window returned a silent 404.
- `publishVersion` and `archiveVersion` were POST stubs that 404'd until Story 3.7 replaced them with PUT `/status`.
- `/api/designer` (no 's') typo had to be fixed in Story 3.2.

### Spec Errors in Dev Notes

Several dev notes prescribed approaches that were wrong at implementation time:
- Story 3.2: API path missing 's' (`/api/designer` vs `/api/designers`)
- Story 3.4: `isValidFieldKey` spec said to place it inside `PropertyInspector.tsx` — triggers react-refresh lint error; had to be extracted to a separate module
- Story 3.7: Single `SaveChangesAsync` for the publish path fails because PostgreSQL evaluates partial unique indexes per-statement — required two-phase commit inside an explicit transaction

### Deferred Circular Dependency

`RepeaterRowDrawer` circular import was documented as a TODO in Story 3.1 and sat for 8 stories before Story 3.9 forced resolution. The risk materialized in Story 3.5 (DynamicComponent wired into preview) but wasn't addressed until Story 3.9.

### Accumulated Lint Debt

26 pre-existing `react-refresh/only-export-components` warnings established in Story 3.1 carried through all 11 stories, never cleaned up.

### Windows-Specific Workaround

vitest's default `forks` pool intermittently fails on Windows with `Timeout waiting for worker to respond`. Required `pool: 'threads'` (Story 3.3). Undocumented at the project level — discoverable only by hitting it.

## Key Insights

1. **A story is done when a user can perform the described action — not when tests pass.** Tests passing is a necessary but not sufficient condition for "done."

2. **Stub code in production is broken code.** When a story ships stub code that intentionally 404s, it must document the known broken state and never be marked "done" without a follow-up verify plan.

3. **Spec divergence should be expected.** Dev notes are proposals, not ground truth. When implementation diverges (two-phase commit, module extraction, API path fix), document the divergence clearly in Completion Notes. The agents did this well — but the process should expect it.

4. **Circular dependency risks should be assessed at story-planning time.** The `RepeaterRowDrawer` circular import was foreseeable in Story 3.1. An architectural review at planning time catches these before they sit as TODOs for 8 stories.
