## Critical Finding (Root Cause of All P1–P7 Issues)

**Stories were accepted when automated tests passed. The application was never verified by running it as a user before marking done across any of the 7 epics.**

jukhan ran the full Aspire stack (`dotnet run --project src/FormForge.AppHost`) and discovered:

| Priority | Issue |
|----------|-------|
| P1 🔴 | Browser refresh → 401 → login page (auth refresh broken) |
| P2 🔴 | Designer UI layout broken (panels overflow, no vertical scroll) |
| P3 🔴 | Designer not creating/updating DB tables |
| P4 🟡 | Menu → sub-menu relationship broken |
| P5 🟡 | Can't assign role or designer to menu item |
| P6 🟡 | Role permission settings UI not visible |
| P7 🟢 | UI look & feel is low-grade (forms, buttons, inputs) |

These bugs exist across Epics 2–5, meaning core functionality was broken throughout the entire polish sprint. The CI gate (unit + integration tests) does not catch real user-flow failures.

## What Went Well

- Infrastructure uplift: Playwright, axe-core CI gate, GitHub Actions pipeline — all net-new and valuable.
- i18n architecture now solid: orphan-detection script, synchronous init, backend ProblemDetails compliance.
- Theme system (client + server persistence + CSP nonce) is well-architected.
- Code review process caught real bugs (double `applyTheme` call, missing `--radius` in slate-dark, aria-grabbed deprecation).
- Team documented deferred items and debug logs in story files — good institutional knowledge trail.

## What Didn't Work

- **No end-to-end manual verification.** The `verify` skill was never used across 42 stories.
- **Polishing a broken foundation.** Epic 7 added UX polish while core menu, designer, and auth flows were broken.
- **Test isolation gave false confidence.** Unit/integration tests passed while user-facing flows failed.
- **Story completion criteria = tests pass**, not "feature works for a user."

---

## Key Lessons

1. Every story must end with a manual smoke test of the happy path in the real running app (`/verify` or equivalent).
2. A "done" story is one where a user can perform the described action, not one where tests pass.
3. Before any future polish/hardening epic, run a full manual regression of all prior epics' core flows.
4. The `verify` skill should be the last step in every `bmad-dev-story` run.
